## vhal aidl适配
2026-03-26
目前代码开发适配代码开发完成，可以正常访问vhal aidl服务 但是有一下两个问题

1.getValue 的回调函数触发慢

2.getValue 有概率拿到空数据

这两个问题会直接导致下线配置出现bug 目前已经跟曾涛同步过

另外代码改动大 需要内部review才能合入
2026-03-26
通过抓取trace发现caraudio收到数据就是慢了 曾涛说是servicemanager慢了 [[trace_analysis.md]]
![[img_v3_02105_6f8b5eab-4ea3-45e7-8985-e42ee9b209dg.jpg]]
最后结论caraudioserver 的main函数写的有问题
```
int main(int argc, char** argv)

{

    ALOGI("CarAudioService <Daemon> begin.....");

    sp<ProcessState> proc(ProcessState::self());

    sp<IServiceManager> sm = defaultServiceManager(); //取得ServiceManager

    //**CarAudioService::instantiate();**

    ProcessState::self()->startThreadPool(); //启动缓冲池
     CarAudioService::instantiate();

    IPCThreadState::self()->joinThreadPool(); //这里是把服务添加到 Binder闭合循环进程中

    ALOGI("CarAudioService <Daemon>  end.....");

}
```
原因是：在startThreadPool之前就开始服务的初始化流程，这导致当前服务仅有一个线程（主线程）。
之前hidl没有问题是因为vhal的get函数是同步的，因此就是按照顺序执行下去的。
但是VHAL aidl getValue是异步的，但是现在服务仅有一个主线程，那么也就是说这个只有在服务初始化完成之后才会释放主线程来接口其他进程的回调函数。因此会完了1s多。
