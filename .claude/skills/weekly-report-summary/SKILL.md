# 周报总结技能

## 技能说明

此技能用于快速总结周报的核心内容，输出简洁的结构化摘要，并原封不动更新日报模板的待办任务。完成后会询问用户是否要上传到 GitHub。

## 使用方法

用户提供周报文件的完整路径，技能将自动读取并总结，然后将下周计划原封不动覆盖到日报模板，最后询问是否上传到 GitHub。

## 执行步骤

### 1. 读取周报文件
使用 Read 工具读取用户指定的文件路径。

### 2. 提取关键信息

从文件中提取以下四类信息：

#### 完成的任务
- 从「待办任务」中找出标记为 `[x]` 已完成的任务
- 从「开发任务」中找出带有日期标记为"今天"或"已完成"的任务
- 记录任务名称、完成状态和完成日期

#### Bug 解决情况
- 从「bug解决」中找出所有 bug 记录
- 记录 bug 编号、问题描述、处理方式
- 统计 bug 总数

#### 技术收获
- 从「今日收获」中提取关键技术知识点
- 格式：`**[关键点]**：[描述]`

#### 下周/明日计划
- 从「待办任务」中找出未完成的任务 `[ ]`
- 从「明日安排」中提取具体计划

### 3. 输出总结

使用结构化表格格式输出：

```markdown
## 📋 周报总结 YYYY-WXX

### ✅ 完成的任务

| 任务 | 状态 | 完成日期 |
|------|------|----------|
| [任务1] | ✅ 已完成 | YYYY-MM-DD |
| [任务2] | ✅ 已完成 | YYYY-MM-DD |

### 🐛 Bug 处理情况（共 N 个）

| Bug 编号 | 问题描述 | 处理方式 |
|----------|----------|----------|
| #XXXXXXX | [简要描述] | [转给XX/已修复/待处理] |
| #XXXXXXX | [简要描述] | [转给XX/已修复/待处理] |

### 📚 技术收获

- **[关键点]**：[描述]
- **[关键点]**：[描述]

### 📅 下周计划

- [ ] [待办任务1]
- [ ] [待办任务2]
```

**格式要求**：
- **表格化展示**：任务和 Bug 使用 Markdown 表格，清晰对齐
- **Bug 数量统计**：在 Bug 部分标题显示总数
- **技术收获**：从「今日收获」中提取关键知识点
- **状态符号**：✅ 已完成、🔄 进行中、📅 已延期

### 4. 更新日报模板（重要）

**必须执行**：将周报中的**完整待办任务列表**覆盖到日报模板的「待办任务」部分。

**模板路径**: `D:\code\文档\模板\周报模板-简洁版.md`

**核心原则**：用周报的完整待办任务列表（包含所有已完成✅和未完成📋的状态）**整体替换**日报模板的待办任务区域。

**操作步骤**:
1. 读取日报模板文件
2. 读取周报文件
3. 从周报中提取 **`## 待办任务`** 部分的**完整内容**（包括已完成和未完成的任务）
4. 用周报中的待办任务列表**整体替换**日报模板中的待办任务
5. **原封不动**保持任务内容、格式、缩进和完成状态

**关键点**：
- **提取完整列表**：不要只提取"下周计划"，而是提取周报中 `## 待办任务` 的**全部内容**
- **保持状态同步**：周报中 `[x]` 已完成的任务要保留， `[ ]` 未完成的也要保留
- **不做任何修改**：不要删除、不要合并、不要更改描述

### 5. 询问是否上传到 GitHub

**必须执行**：在完成所有步骤后，询问用户是否需要上传到 GitHub。

**询问方式**：
```
---

📦 是否需要上传到 GitHub？

请回复：
- "是" 或 "上传" → 执行上传
- "否" 或 "不上传" → 跳过上传
```

**等待用户回复后**：
- 如果用户确认「是」→ 调用 git-upload skill 执行上传
- 如果用户选择「否」→ 结束技能，输出完成信息

## 输出要求

- 每项内容保持简短，一句话概括
- **优先使用表格格式**，比纯列表更清晰易读
- Bug 编号使用加粗格式
- 保持格式统一规范
- **必须原封不动覆盖日报模板，不要做任何改动**
- **必须询问用户是否上传 GitHub，得到确认后才能执行上传**

## 注意事项

- 如果某个部分没有内容，该部分可以省略
- 如果文件不存在或无法读取，提示用户检查路径
- 优先提取明确标记为"已完成"的任务，避免误判
- **日报模板更新失败时，必须向用户报告错误**
- **重要：只需要覆盖内容，不要修改、删除或添加任何任务**
- **必须等待用户明确确认后才能执行 GitHub 上传**
