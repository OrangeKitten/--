# AudioPolicyMix与SwAudioOutputDescriptor关系

从**dump APM**中可知**Audio Policy Mix** 信息如下
```  
Audio Policy Mix 6 (0xb4000078516cce10):

  - mix type: MIX_TYPE_PLAYERS

  - Route Flags: MIX_ROUTE_FLAG_RENDER

  - device type: AUDIO_DEVICE_OUT_BUS

  - device address: BUS06_NOTIFICATION_OUT

  - output: 85

    - Criterio#n 0: RULE_MATCH_ATTRIBUTE_USAGE AUDIO_USAGE_ALARM

    - Criterion 1: RULE_MATCH_ATTRIBUTE_USAGE AUDIO_USAGE_NOTIFICATION

    - Criterion 2: RULE_MATCH_ATTRIBUTE_USAGE AUDIO_USAGE_ASSISTANCE_SONIFICATION
```
从该信息可以知道输出流是85.
```
- Output 85 dump:

 Latency: 75

 Flags 00000000

 ID: 28

 Sampling rate: 48000

 Format: 00000001

 Channels: 00000003

 Devices: {type:0x1000000,@:BUS06_NOTIFICATION_OUT}
```

==那么有个疑问这个85是怎么跟AudioPolicyMix绑定在一起的呢?==
==在创建动态路由的时候并没有设置这个outputid。==

在文件**AudioPolicyManager.cpp**中存在如下代码
```cpp
status_t

AudioPolicyManager::registerPolicyMixes(const Vector<AudioMix> &mixes) {
···
// 从 mix 配置中获取设备地址（如蓝牙地址、USB设备路径等）
String8 address = mix.mDeviceAddress;
// 获取设备类型（如 AUDIO_DEVICE_OUT_SPEAKER、AUDIO_DEVICE_OUT_BLUETOOTH_A2DP 等）
audio_devices_t type = mix.mDeviceType;
ALOGV(" registerPolicyMixes() mix %zu of %zu is RENDER, dev=0x%X addr=%s",i, mixes.size(), type, address.string());
// 从硬件模块中查找匹配的设备描述符
// 参数：设备类型、设备地址、设备名称（空）、音频格式（默认）
sp<DeviceDescriptor> device = mHwModules.getDeviceDescriptor(
    mix.mDeviceType, mix.mDeviceAddress, String8(), AUDIO_FORMAT_DEFAULT);
if (device == nullptr) {
  // 如果找不到对应的设备描述符，注册失败
  res = INVALID_OPERATION;
  break;
}

bool foundOutput = false;
// 第一步：尝试在已经打开的输出流中找到支持该设备的输出
// 遍历所有现有的输出流描述符
for (size_t j = 0; j < mOutputs.size() && !foundOutput && res == NO_ERROR;
     j++) {
  sp<SwAudioOutputDescriptor> desc = mOutputs.valueAt(j);

  // 检查该输出流是否满足以下条件：
  // 1. 不是复制输出（isDuplicated() == false）
  //    - 复制输出是指同时输出到多个设备的特殊输出流
  // 2. 该输出流支持的设备列表中包含目标设备
  if (!desc->isDuplicated() &&
      desc->supportedDevices().contains(device)) {
    // 尝试将 mix 注册到找到的输出流描述符
    if (mPolicyMixes.registerMix(mix, desc) != NO_ERROR) {
      // 注册失败，记录错误日志
      ALOGE("Could not register mix RENDER,  dev=0x%X addr=%s", type,
            address.string());
      res = INVALID_OPERATION;
    } else {
      // 注册成功，标记已找到输出流
      foundOutput = true;
    }
  }
}
```
这么看在注册AudioPolicyMix的时候会轮询outputs，绑定一个output，这个output要求device与AudiopolicyMix的device相匹配并且不是isDuplicated。 那么我们再看一下**audio_policy_configuration.xml**
```
 <route type="mix" sink="Sys Notification Bus"                     sources="sys_notification,carplay_8k_sys,carplay_16k_sys,carplay_24k_sys,carplay_32k_sys,carplay_48k_sys,mmap_no_irq_out"/>
```
有多个*output = source*   路由到*device = Sys Notification Bus* .那么结合上面的匹配规则这么看Audio Policy Mix 选择output = 85,是因为在route中sys_notification 在source的第一个。
